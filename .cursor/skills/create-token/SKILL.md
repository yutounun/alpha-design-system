# Create design token

## When to use

Use when editing [`src/styles/theme.css`](../../src/styles/theme.css) to introduce or adjust semantic colors, borders, fills, rings, or interaction states.

## Rules (short)

1. **No `color-mix()`** — Use explicit `hex` or `rgba()` / `hsla()` values only. If you need a translucent border or hover wash, define a **named** token; do not derive it with `color-mix()` in CSS. See also [guardrail §2.1](../guardrail.md).

2. **Follow existing families** — Prefer the same semantic prefixes as tokens already in `@theme`:
    - Borders: `--color-border-high`, `--color-border-medium`, `--color-border-low` → add e.g. `--color-border-danger`, `--color-border-invalid`, `--color-border-*-hover` / `-focus` when needed.
    - Fills: `--color-fill-*` → add e.g. `--color-fill-danger-hover` for destructive hover surfaces.
    - Focus / rings: `--color-focus`, `--color-focus-ring`, `--color-focus-ring-soft`, `--color-danger-ring`, `--color-invalid-ring` — keep ring/border roles separate from solid fills.

3. **Light and dark** — If a token is overridden in `.dark`, add matching entries there with values tuned for dark surfaces (do not rely on `color-mix()` for theme switching).

4. **Consumers** — Point components at Tailwind utilities backed by these variables (`border-border-danger`, `bg-fill-danger-hover`, `ring-danger-ring`, etc.). Avoid `dark:` in `className` when tokens already retoken in `.dark`.

## After you change tokens

Regenerate the agent token dictionary when present:

```bash
python3 scripts/build-design-tokens-json.py
```
