このページは、リポジトリ全体をシニア開発者の視点で俯瞰したときの**評価**と、優先度付きの**改善の方向性**をまとめた内部向けメモです。採用者向けの「使い方」は Getting Started 配下の各ページを参照してください。

## 総評

規模は控えめながら、**公開コンポーネントライブラリとして必要な意思決定が揃っている**リポジトリです。npm と shadcn レジストリの二系統配布、Changesets、トークン中心のスタイル、Storybook 10 とアクセシビリティアドオン、Vitest（ブラウザ）とインタラクションテスト（`play`）まで一通りあり、**オープンなデザインシステムとして十分戦える土台**があります。

## 強み（維持したい点）

1. **配布戦略** — `package.json` の `exports`、スタイルのエントリ（`styles.css`）、`scripts/build-registry.mjs` によるレジストリ・バレル・エクスポートの同期は、運用負荷を抑えつつ一貫性を保ちやすいです。
2. **ドキュメントと体験の一体** — README で Storybook を入口に据え、MDX とトークンギャラリーを同じサーフェスに載せる方針は、初見の開発者の理解コストが低いです。
3. **ビルドパイプライン** — Vite のライブラリビルド、`preserveModules`、外部依存の切り出し、`vite-plugin-dts`、`sideEffects` で CSS を明示するなど、ライブラリ配布として妥当な選択が多いです。
4. **品質ゲート** — CI で lint、型チェック、Playwright 前提の Storybook / Vitest、`build:lib` まで回しており、PR 時の最低ラインは取れています。
5. **コンポーネント配置** — `src/components/<name>/` に実装・`index`・stories・mdx が揃うレイアウトは、チーム拡張時も説明しやすいです。

## 改善余地（優先度つき）

### 高：CI と本番に近いビルドのギャップ

Pull Request の CI は `pnpm run build:lib` までです。README にある `pnpm run build`（レジストリビルドと Storybook の静的ビルドを含む）までは踏んでいません。レジストリ生成や Storybook ビルドだけが壊れる不具合は、マージ後まで気づきにくいので、**`build` 相当を CI に追加する**、または **main 向け・定期ジョブ**でカバーすると安心です。

### 高：テストの分布

`*.test.ts` のようなユニットテストはほぼ置かず、**ストーリーの `play` を中心としたテスト**に寄っています。デザインシステムとしては十分あり得る一方、次のような層は**軽いユニットテストやスナップショット**があると回 regressions が楽になります。

- `src/lib/utils.ts` などの純関数
- `src/docs/token-display/parse-theme-tokens.ts` のようなパーサ
- レジストリ生成スクリプトの契約（`exports` の形など）

### 中：巨大なストーリーファイル

`src/pages/dashboard/dashboard.stories.tsx` は行数が非常に多く、デモとしての価値は高い一方、**レビュー・差分追跡・Storybook の読み込み**のコストが上がります。サブストーリーへの分割や、プレゼンテーション用の薄いラッパへの切り出しを検討すると長期的に楽です。

### 中：ESLint と `eslint-config-next`

`eslint.config.mjs` は Next.js のプリセットを利用しています。本体は Vite と Storybook が中心のため、**ルールの一部がプロダクトとズレる**可能性はあります。害が出ていなければ維持でもよいですが、Next を併用しない方針なら、**React + TypeScript + Storybook 向けのフラット構成**への寄せも選択肢です。

### 中：Node のバージョン方針

CI と Release は Node 20 固定です。エコシステムの推奨 LTS が上がるにつれ、**ローカルと CI の差**やネイティブ依存の挙動差が出やすいので、`.node-version` や `package.json` の `engines` で期待バージョンを明示すると、メンテナと採用者の双方に親切です。

### 低：コントリビューション用の短いガイド

リポジトリ直下に `CONTRIBUTING.md` がない場合、外部からの PR では「コンポーネント追加手順」「Changeset の要否」「CI で何が走るか」が散らばって見えます。README の要点へのリンクだけでもよいので、**1 ページにまとめる**とよいです。

## まとめ

アーキテクチャ（Radix、トークン、二重配布、レジストリ自動同期）は成熟しており、**次のレバーは CI のビルド範囲の拡大、ロジック層のユニットテスト、巨大デモの分割**が投資対効果が高いです。
